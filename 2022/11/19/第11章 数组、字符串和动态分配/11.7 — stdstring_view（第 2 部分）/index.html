11.7 — std::string_view（第 2 部分）

发表于 2022-11-19 更新于 2022-11-26 分类于 C++

在上一课中，我们讨论了 C 风格的字符串，以及使用它们的危险。C 风格的字符串速度很快，但它们不如std::string.

但是std::string（我们在第 4.17 课– std::string 简介中介绍过）有其自身的一些缺点，尤其是涉及到 const 字符串时。

在内部，main 将字符串“hello”复制了 3 次，得到 4 个副本。 首先是字符串字面量“hello”，它在编译时已知并存储在二进制文件中。 当我们创建 char[] 时创建一个副本。 以下两个 std::string 对象各自创建一个字符串副本。

那么为什么 std::string 会复制用于初始化它的字符串呢？ std::string 为它管理的字符串数据提供了一些有用的保证：

  • 只要字符串对象还活着，字符串数据就会有效，并在字符串对象死亡时被清理。
  • 字符串的值只能由字符串对象修改。
  • 可以修改字符串的值而不影响其他对象（对于非常量字符串）。

std::string 对象无法控制使用哪种字符串来初始化它（这是调用者的责任）——调用者可以在 std::string 对象初始化后立即销毁或修改初始化字符串。 初始化后，字符串对象不能以任何方式依赖初始化值，否则可能违反上述保证。

通过复制初始化字符串（只有它可以访问），字符串对象可以确保字符串数据的值和生存期独立于其他对象（包括初始化字符串）。

然而，在某些情况下，我们不需要这些好处（特别是对于 const std::string 对象，它们不能修改它们的值）。 考虑这样一种情况，我们知道 std::string 的寿命不会超过它的初始化字符串，并且初始化字符串不会被修改。 在这种情况下，我们为不需要的好处（独立于初始化字符串）付出了高昂的代价（制作初始化字符串的副本）。

介绍 std::string_view

考虑一下您家中的一扇窗户，看着停在街上的汽车。 您可以透过窗户看到汽车，但不能触摸或移动汽车。 你的窗户只是提供了汽车的视野，这是一个完全独立的对象。

C++17 引入了另一种使用字符串的方法，std::string_view，它位于 标头中。

与保留自己的字符串副本的 std::string 不同，std::string_view 提供在别处定义的字符串的视图（view）。

std::string_view 不仅速度快，而且具有我们从 std::string 中了解到的许多功能。

因为 std::string_view 不创建字符串的副本，所以如果我们更改查看的字符串，更改会反映在 std::string_view 中。

我们修改了 arr，但 str 似乎也在发生变化。 那是因为 arr 和 str 共享它们的字符串。 当您使用 std::string_view 时，最好避免在 std::string_view 的剩余生命周期内修改底层字符串，以防止混淆和错误。

最佳实践：

当您只需要对字符串进行只读访问时，首选 std::string_view 而不是 std::string 和 C 风格的字符串（除非您不能保证正在查看的字符串在 std::string_view 的生命周期内保持活动状态，在这种情况下，您应该更倾向使用 std::string)。

通过 const std::string& 或 std::string_view 传递字符串？

一个经常出现的问题：通过 const std::string& 或 std::string_view 传递字符串更好吗？

如果我们想写一个接受字符串参数的函数，使参数成为 std::string_view 是最灵活的选择，因为它可以高效地处理 C 风格的字符串参数（包括字符串文字）、std::string 参数（这将隐式转换为 std::string_view) 和 std::string_view 参数：

请注意，我们通过值而不是 const 引用传递 std::string_view。 这是因为 std::string_view 通常可以快速复制，而按值传递是廉价复制类型的最佳选择。

在一种情况下，将参数设为 const std::string& 通常会更好：如果您的函数需要调用其他采用 C 风格字符串或 std::string 参数的函数，那么 const std::string& 可能是更好的选择，因为不能保证 std::string_view 以 null 终止（C 风格字符串函数期望的东西）并且不能有效地转换回 std::string。

最佳实践：

优先使用 std::string_view（按值）而不是 const std::string& 传递字符串，除非您的函数调用其他需要 C 风格字符串或 std::string 参数的函数。

作者的话：

未来课程中的许多示例都是在引入 std::string_view 之前编写的，并且在应该首选 std::string_view 时仍然使用 const std::string& 作为函数参数。 我们正在努力清理这些。

所有权问题

std::string_view 的生命周期独立于它正在查看的字符串（意味着正在查看的字符串可以在 std::string_view 对象之前销毁）。 如果发生这种情况，那么访问 std::string_view 将导致未定义的行为。

std::string_view 正在查看的字符串必须在其他地方创建。 它可能是一个与程序一样长的字符串文字，或者是一个 std::string，在这种情况下，字符串一直存在，直到 std::string 决定销毁它或 std::string 死亡。

std::string_view 不能自己创建任何字符串，因为它只是一个视图。

在 main 中访问返回的 std::string_view 会导致未定义的行为，这在作者的机器上会产生奇怪的字符。

当我们从 std::string 创建 std::string_view 然后修改 std::string 时，也会发生同样的情况。 修改std::string可能会导致其内部字符串消失并在不同位置被新字符串替换。 std::string_view 仍会查看旧字符串所在的位置，但它不再存在了。

警告：

确保使用 std::string_view 查看的底层字符串不会超出范围并且在使用 std::string_view 时不会被修改。
