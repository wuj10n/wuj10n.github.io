什么是用户定义/程序定义类型？

回到上一章的介绍（9.1 - 复合数据类型介绍），我们介绍了想要存储分数的挑战，分数的分子和分母在概念上是联系在一起的。在那节课中，我们讨论了使用两个单独的整数来独立存储分数的分子和分母的一些挑战。

如果 C++ 有一个内置的分数类型，那将是完美的——但事实上没有。还有数百种 C++ 未包含的潜在有用的类型，因为C++不可能预测可能需要的一切类型（更不用说实现和测试这些东西了）。

相反，C++ 以不同的方式解决此类问题：允许我们创建全新的自定义类型以用于我们的程序！此类类型通常称为用户定义类型（user-defined types）（尽管我们认为术语程序定义类型（program-defined types）更好——我们将在本课稍后部分讨论两者之间的区别）。C++ 有两类复合类型允许这样做：枚举类型（包括无作用域和作用域枚举）和类类型（包括结构、类和联合）。

命名程序定义的类型

按照惯例，程序定义的类型以大写字母开头并且不使用后缀（例如Fraction, not fraction, fraction_t, 或Fraction_t）。

最佳实践：

以大写字母开头命名程序定义的类型，不要使用后缀。

在整个多文件程序中使用程序定义的类型

每个使用程序定义类型的代码文件在使用之前都需要查看完整的类型定义。预先声明是不够的。这是必需的，以便编译器知道为该类型的对象分配多少内存。

为了将类型定义传播到需要它们的代码文件中，程序定义的类型通常在头文件中定义，然后#included 到需要该类型定义的任何代码文件中。这些头文件通常被赋予与程序定义类型相同的名称（例如，名为 Fraction 的程序定义类型将在 Fraction.h 中定义）

最佳实践：

仅在一个代码文件中使用的程序定义类型应在该代码文件中定义为尽可能靠近第一个使用点。

在多个代码文件中使用的程序定义类型应在与程序定义类型同名的头文件中定义，然后根据需要#included 到每个代码文件中。

类型定义部分免于单一定义规则

在第2.7课- 前向声明和定义中，我们讨论了单一定义规则如何要求每个函数和全局变量在每个程序中只能有一个定义。要在不包含定义的文件中使用给定的函数或全局变量，我们需要一个前向声明（我们通常通过头文件传播）。这是可行的，因为当涉及到函数和非 constexpr 变量时，声明足以满足编译器的要求，然后链接器可以将所有内容连接起来。

但是，以类似的方式使用前向声明不适用于类型，因为编译器通常需要查看完整定义才能使用给定类型。我们必须能够将完整的类型定义传播到每个需要它的代码文件。

为此，类型部分免于单一定义规则：允许在多个代码文件中定义给定类型。

您已经使用了这种能力（可能没有意识到）：如果您的程序有两个代码文件，这两个代码文件都是#include <iostream>，您正在将所有输入/输出类型定义导入到这两个文件中。

有两个警告值得了解。首先，每个代码文件仍然只能有一个类型定义（这通常不是问题，因为标头保护会阻止这种情况）。其次，给定类型的所有类型定义必须相同，否则将导致未定义的行为。

命名法：用户定义类型与程序定义类型(user-defined types vs program-defined types)

“用户定义类型”一词有时会出现在随意的谈话中，在 C++ 语言标准中也会被提及（但未定义）。在随意的谈话中，该术语往往表示“您自己定义的类型”（例如上面的 Fraction 类型示例）。有时这还包括类型别名。

但是，正如在 C++ 语言标准中所使用的那样，用户定义类型是指未定义为核心 C++ 语言的一部分的任何类型（换言之，非基本类型）。因此，C++ 标准库中定义的类型（例如std::string）在技术上被认为是用户定义的类型，您自己定义的任何类型也是如此。

为了提供额外的区别，C++20 语言标准定义了术语“程序定义的类型”，仅表示您自己定义的类型。在谈论此类类型时，我们会更喜欢这个术语，因为它不那么含糊。

Type Meaning Examples
Fundamental 内置于核心 C++ 语言中的类型 int, std::nullptr_t
User-defined 非基本类型
（随意使用，通常用于表示程序定义的类型）		 std::string, Fraction
Program-defined 自己定义的类类型或枚举类型 Fraction
